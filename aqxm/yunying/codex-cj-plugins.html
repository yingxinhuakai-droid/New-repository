插件详解

Computer Use已启用

用于控制 Mac 上的本地应用。它能看应用窗口、点击、输入、滚动、切换界面。

适合你的场景

  • 操作安全工具界面，导出扫描报告。
  • 打开本地文档、表格、浏览器或编辑器。
  • 完成重复点击、复制、粘贴、上传、下载。
通用必备本地软件
Chrome已启用

用于控制你的 Chrome 浏览器，特别适合需要已有登录状态的网站。

适合你的场景

  • 访问测试后台、客户系统、管理平台。
  • 检查页面展示、登录流程、按钮功能。
  • 辅助 Web 安全测试前的信息整理。
Web 测试登录态
Spreadsheets已启用

用于创建、编辑、分析 Excel、CSV、TSV 等表格文件。

适合你的场景

  • 制作资产清单、整改跟踪表。
  • 按风险等级、系统、负责人做统计。

一次完整操作指引：从扫描结果到汇报材料

目标：假设你拿到一份漏洞扫描结果，要把它整理成漏洞台账、生成统计结论，再做成汇报 PPT。这是安全工作里很常见的一条完整流程。

准备文件

把扫描结果放到当前项目文件夹里，比如 CSV、Excel、HTML 报告、PDF 报告都可以。你可以直接告诉 Codex 文件名和你想要的结果。

帮我读取这个扫描结果，整理成漏洞台账，字段包括：资产、漏洞名称、风险等级、影响说明、修复建议、负责人、整改状态。

用 Spreadsheets 整理台账

让 Codex 把原始结果变成结构清晰的表格，按风险等级上色，增加统计页和整改跟踪字段。

把高危、中危、低危分开统计，并生成一个“整改跟踪”工作表，默认状态为待整改。

用 Data Analytics 找结论

如果数据量大，可以进一步分析哪些系统风险最高、哪些漏洞重复出现、整改优先级应该怎么排。

分析这份漏洞台账，告诉我前三个风险最高的系统、最常见的漏洞类型，以及优先整改建议。

用 Chrome 或 Computer Use 查证页面

如果需要确认某个漏洞页面是否还能访问，或者要查看后台状态，可以让 Codex 使用 Chrome 或本地软件协助检查。

打开这个测试系统后台，帮我确认这些资产是否仍然可访问，并把结果补充到表格中。

用 Presentations 生成汇报 PPT

最后把表格结论做成汇报材料：项目概览、风险分布、重点漏洞、整改计划、下一步建议。

根据漏洞台账生成一份 8 页安全汇报 PPT，面向管理层，重点突出风险、影响和整改优先级。

用 Gmail、Notion 或 Linear 做跟进

如果你启用了对应插件，可以把结论发邮件、写入知识库，或创建整改任务。

根据这份整改清单，帮我起草一封发给研发负责人的整改通知邮件。

按你的安全工作场景怎么选插件

场景一：漏洞扫描结果整理

推荐插件：Spreadsheets、Data Analytics、Presentations。

  • 把扫描工具导出的 CSV 或 Excel 变成漏洞台账。
  • 自动统计高危、中危、低危数量。
  • 输出整改计划和汇报 PPT。

场景二：Web 系统检查和后台操作

推荐插件：Chrome、Computer Use。

  • 打开测试系统、客户后台、管理平台。
  • 检查页面流程、表单、按钮和登录状态。
  • 把观察到的内容整理成测试记录。

场景三：代码安全审查

推荐插件：GitHub、Chrome、Data Analytics。

  • 查看代码变更和 Pull Request。
  • 分析是否存在鉴权、注入、越权、敏感信息泄露等风险。
  • 把发现的问题整理成修复建议。

场景四：日志和攻击线索分析

推荐插件：Data Analytics、Spreadsheets。

  • 从访问日志中识别异常 IP、路径、时间段。
  • 统计攻击类型和高频目标。
  • 形成事件时间线和处置建议。

场景五：安全报告和培训材料

推荐插件：Presentations、Spreadsheets、Creative Production。

  • 做渗透测试汇报、整改复盘、培训课件。
  • 把技术风险转成清晰图表。
  • 生成封面、课程视觉或宣传图。

场景六：项目管理和整改跟踪

推荐插件：Notion、Linear、Gmail、Slack。

  • 把漏洞分配给负责人。
  • 跟踪整改状态和截止时间。
  • 整理会议纪要、邮件和团队沟通。

常用提问模板

你可以直接复制这些说法来用

1. 帮我把这个扫描结果整理成漏洞台账，按风险等级分组，并生成整改跟踪表。 2. 分析这份漏洞台账，告诉我当前最需要优先处理的 5 个风险点。 3. 根据这个 Excel 生成一份安全汇报 PPT，面向领导，控制在 8 页以内。 4. 打开这个测试系统，帮我检查登录、列表、详情、导出这几个流程是否正常。 5. 帮我审查这个 GitHub PR，重点看鉴权、输入校验、敏感信息泄露和依赖风险。 6. 根据这批访问日志，分析是否存在扫描、爆破、目录探测或异常请求。 7. 根据整改清单，帮我写一封通知研发负责人整改的邮件，语气专业但不要太强硬。 8. 把这次项目经验整理成 Notion 知识库文档，包括背景、问题、处理过程、复盘和可复用模板。

建议使用顺序：先掌握 Computer Use、Chrome、Spreadsheets、Presentations；再根据实际工作启用 GitHub、Data Analytics、Notion、Gmail、Linear。
